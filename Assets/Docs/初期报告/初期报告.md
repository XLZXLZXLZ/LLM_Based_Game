第一章 项目主要内容 
1.1 项目背景 
现代电子游戏中，非玩家角色（NPC）作为构建世界观、推进剧情叙事与提供
交互体验的核心载体，其智能化水平已成为衡量游戏沉浸感与玩家体验的重要指
标。传统游戏NPC主要依赖预编写的对话树和有限状态机实现交互逻辑，这种设
计范式虽然保证了内容可控性，但也导致NPC行为模式单一、对话选项有限、难
以响应玩家的非预设输入。与此同时，游戏内容的生成与迭代同样高度依赖开发
者的预先设计，玩家作为体验者难以参与到游戏机制本身的创作过程中。随着玩
家对开放世界、涌现式叙事和用户生成内容（UGC）等游戏体验需求的不断增长，
传统游戏系统在交互自由度与内容扩展性上的局限日益凸显。 
大语言模型（Large Language Model, LLM）及LLM-Agent 技术的快速发展
为上述问题提供了新的解决路径。以GPT、Claude等为代表的LLM展现出强大的
自然语言理解、开放域对话、角色扮演以及代码生成能力。在此基础上，LLM-Agent
作为新兴技术范式，通过赋予大语言模型感知、规划、记忆与行动能力，使其能
够作为自主智能体参与复杂任务的执行。这一技术进展不仅为构建能够动态响应
玩家行为的智能NPC提供了可能，也为通过自然语言描述直接生成游戏逻辑脚本
——例如玩家以自然语言自定义游戏逻辑并由系统自动转化为可执行代码——
开辟了新的技术空间。 
当前，以AI伴侣、虚拟角色对话为代表的应用已初步验证了LLM在角色扮
演场景中的可行性，用户可与AI驱动的虚拟角色进行自由对话交互。然而，这
类应用普遍缺乏游戏机制的支撑，对话内容不与游戏规则、数值系统产生实质性
关联，本质上仍属于开放式聊天而非完整的游戏体验。同时，LLM所具备的代码
生成能力尚未被有效引入游戏内容的动态创作流程。如何将LLM的自然语言能力
与结构化的游戏机制相融合，使玩家的自然语言输入既能驱动NPC交互与剧情演
进，又能转化为符合游戏规则的可执行脚本，是当前LLM融合游戏领域亟待探索
的方向。 
1.2 项目指标 
指标一：LLM驱动的NPC智能体系统。  
构建基于大语言模型的NPC对话与决策系统，实现NPC与玩家的自由对话交
互。设计长期记忆与上下文管理机制，支持印象管理、好感度等交互数值的动态
维护。NPC具备对玩家游戏行为的感知能力，并能基于对话内容与感知信息自主
生成行为响应，实现从被动应答到主动交互的智能化跃升。 
指标二：自然语言-游戏语义双向转化系统。  
设计自然语言与游戏指令之间的双向转化机制。正向转化支持将玩家的自然
语言输入解析为符合游戏规则的 API 调用与行为指令；逆向转化支持将游戏状
态、事件等结构化信息转化为LLM可理解的语义描述。构建完整的接口层，实现
NPC 智能系统与游戏逻辑系统的有效桥接。 
指标三：六边形网格战棋系统。  
实现基于六边形网格的地图系统，支持地图的生成与管理。集成A等寻路算
法，实现角色在网格地图上的自动路径规划。设计回合制战棋移动机制，支持移
动范围计算、地形影响等策略要素，为游戏提供战棋玩法的基础框架。 
指标四：卡牌战斗系统与LLM动态卡牌生成。  
设计原创的回合制卡牌战斗机制，构建基于 Lua 脚本的卡牌效果执行引擎，
支持灵活的卡牌逻辑定义与运行。引入LLM驱动的动态卡牌生成功能，玩家可通
过自然语言描述卡牌效果，系统自动生成对应的可执行Lua脚本，实现游戏内容
的用户参与式创作。 
指标五：完整可运行的游戏原型。  
整合上述各系统，构建完整可游玩的游戏Demo。包含一定体量的游戏内容，
涵盖关卡设计与剧情流程。完善游戏表现层资源，包括角色动画、UI界面、音效
与背景音乐等，确保游戏原型在功能完整性与体验完成度上达到可演示、可评估
的水平。 
第二章 实施方案简述 
本项目对项目的组织与实施进行了系统性规划。在软件开发层面，项目包含
两大核心系统：Human-Like NPC交互系统与自然语言驱动的GamePlay系统。 
2.1 智能化NPC交互系统 Human-Like NPC 
简单而言，Human-Like NPC的设计理念就是“尽可能还原出更多人类行为”
的NPC，在本项目中，我们将一个标准的NPC分为了四个模块 
NPC 交互系统采用四模块架构设计，分别为INFO、MEMORY、AGENT与PLANNER。 
其中，INFO和MEMORY用于优化LLM的输入结构，使其更准确且真实 
AGENT 与PLANNER 用于扩大LLM的输出影响力，使其不局限于文字输出 
Figure 1 NPC 交互系统概述 
2.1.1 INFO 
INFO 模块作为NPC对话系统的信息基础层，承担着为智能体提供对话语境参
考的核心职能。该模块在架构设计上采用静态与动态相结合的双层信息组织策略。 
静态信息层涵盖预先定义的角色背景叙事、人格特质、价值取向等要素，这
些属性在游戏进程中保持恒定，共同构成角色的核心人格基底，确保NPC在不同
对话场景下能够维持一致的身份认同与行为逻辑。 
动态信息层则负责追踪并记录角色在游戏世界中的交互经历——当玩家执
行各类行为操作时，系统将以结构化日志的形式将相关事件同步至处于感知范围
内的NPC记录中，从而使智能体具备对游戏世界状态变化的感知与响应能力。此
外，考虑到传统对话系统中角色态度倾向过于固化的局限性，本项目引入了情绪
状态字段作为对话生成的调节变量，用以动态修饰 NPC 在特定语境下的态度表
达，从而增强对话交互的自然性与情境适应性。 
Figure 2 INFO 对话信息模块 
2.1.2 MEMORY 
MEMORY 模块的设计立足于对大语言模型上下文窗口容量约束的系统性考量。 
从技术本质而言，任何依赖单一上下文窗口完整承载全部交互细节的方案均
存在固有局限性；记忆机制的核心价值在于使有限的上下文容量能够高效获取并
整合关键信息，而非追求信息的穷尽式存储。 
基于上述认识，本项目借鉴认知科学领域中关于人类记忆系统的经典理论框
架，即瞬时记忆、短期记忆与长期记忆的层级划分，构建了三层递进式记忆架构。 
在该架构下，一个完整的记忆上下文窗口由三个功能层次共同构成： 
其一为当前对话会话的即时上下文，承载正在进行的交互信息。 
其二为近期交互历史的压缩摘要，保留中等时间跨度内的关键语义信息。 
其三为从长期记忆库中基于相关性检索提取的历史条目，提供跨会话的持久
性知识支撑。 
当单次对话会话结束时，系统将触发自动化总结机制，对会话内容进行语义
压缩与关键信息抽取，并将处理结果归档至相应的记忆层级，从而实现记忆信息
在不同时间尺度上的有序流转与持久化存储。 
Figure 3 RAG 拓展LLM上下文能力 
2.1.3 AGENT 
AGENT 模块的设计目标在于突破传统NPC对话系统的功能边界，使智能体不
仅具备自然语言交互能力，更能够通过对话行为对游戏世界产生实质性影响，从
而实现从"被动应答者"向"主动参与者"的角色跃迁。为实现上述目标，本项目构
建了一套基于符号映射的行为调用机制。 
具体而言，系统预先定义了一组自然语言描述与游戏引擎原生接口之间的映
射关系，形成结构化的"API词典"。在对话生成过程中，大语言模型依据交互语
境与玩家请求，在输出自然语言回复的同时，按照预设格式生成相应的伪指令序
列。 
以玩家向NPC提出"给我一个苹果"的请求为例，模型将在生成符合角色人设
的对话回应之余，同步输出与物品赠予行为对应的结构化指令。随后，游戏运行
时环境对该伪指令字符串进行解析与验证，并将格式正确的指令转译为游戏引擎
层面的原生API调用，最终驱动游戏世界状态的实际变更。通过这一机制，本项
目实现了大语言模型驱动的智能体对游戏世界的闭环交互能力，使NPC的对话行
为能够产生超越文本层面的实际游戏效果 
Figure 3 Agent 拓展NPC 交互能力 
2.1.4 PLANNER 
PLANNER 模块作为本项目智能体架构的最终组成部分，旨在为NPC提供基于
环境感知的自主决策能力，其核心设计理念遵循"感知—决策"的认知处理范式。
在传统对话系统中，智能体的响应生成往往直接依赖于输入信息与预设规则，缺
乏对游戏世界当前状态的动态感知能力。 
针对这一局限性，PLANNER 模块引入了信息获取与行为规划相分离的双阶段
处理机制。在感知阶段，模块支持智能体通过两种途径获取游戏环境信息： 
其一为被动接收系统推送的事件日志。 
其二为主动调用信息查询类API以检索特定的游戏逻辑状态。 
在此基础上，决策阶段将整合上述环境信息与当前交互上下文，经由大语言
模型的推理能力生成具有情境适应性的行为规划，并输出相应的执行指令。通过
这一"先感知、后决策"的闭环架构，PLANNER模块赋予了NPC超越简单条件反射
式应答的高阶认知能力，使其能够在复杂动态的游戏环境中展现出更具智能性与
自主性的行为表现。

语言驱动
本项目进一步探索了自然语言处理技术在核心游戏机制层面的深度应用。本
节所述系统的核心目标在于实现自然语言与游戏形式语言之间的双向语义转换，

对游戏可交互内容产生实质性影响。 
更进一步地，该系统超越了静态内容的语义解析范畴，探索了基于自然语言
输入的动态逻辑生成能力。在具体实现层面，本项目以卡牌对战作为基础游戏循
环的载体，构建了一套支持玩家通过结构化自然语言描述动态生成游戏卡牌、并

2.2.2 自然语言编辑卡牌 
在功能定位上，本项目追求的并非简单的预定义API调用，而是探索更具创
造性的深层能力：使玩家能够通过自然语言描述"定义"所期望的卡牌行为逻辑，
即驱动大语言模型直接生成可执行的卡牌逻辑代码，而非仅作为API选择器进行
函数调用的组合。 
这一设计理念借鉴了算法题目评测平台的范式——系统将游戏战斗场景上
下文、角色实体类、视觉特效接口等要素作为形式参数，以类似函数签名的形式
提供给大语言模型，由其根据玩家的自然语言描述生成具体的卡牌行为实现代码。
生成的代码以文本形式持久化存储于本地，并在战斗流程中被动态加载与执行。 
然而，这一方案在Unity引擎环境下面临特定的技术约束。由于C#作为编译
型语言，其运行时并不原生支持动态代码的即时插入与执行，直接生成 C#脚本
的路径在技术层面缺乏可行性。针对这一问题，本项目采用了Lua脚本语言作为
替代方案。Lua作为解释型语言，具备运行时动态加载与执行的能力，且Unity
引擎通过第三方桥接库可良好支持Lua的嵌入式运行环境。 
基于此，本项目通过属性暴露机制将游戏战斗场景对象与相关API接口导出
至Lua运行时环境，使大语言模型能够直接生成符合规范的Lua代码并实现即插
即用。在具体实现架构上，动态生成的Lua卡牌遵循统一的模板规范，继承自标
准卡牌基类并通过Lua函数对基类中定义的功能接口进行覆写。对于大语言模型
的代码生成任务而言，其核心职责即在于根据玩家的自然语言描述，合理实现这
些覆写函数的具体逻辑。

2.2.3 通用LLM 交互框架 
除上述核心功能模块外，本项目还构建了一套面向大语言模型交互的通用服
务接口层。考虑到游戏系统中存在多种异构的LLM调用场景——涵盖NPC对话响
应生成、行为决策推理、动态内容扩展等不同类型的任务需求，若各模块独立实
现通讯逻辑，将导致代码冗余与维护成本的显著上升。为此，本项目在架构层面
设计了统一的LLMManager 管理器组件，作为游戏系统与大语言模型服务之间的
中央通讯枢纽。该管理器封装了请求队列调度、会话上下文管理、异步响应处理
等通用功能，为游戏内各功能模块提供标准化的LLM调用接口，从而实现了模型
交互逻辑的解耦与复用，有效提升了系统的可维护性与可扩展性。 
第三章 可行性研究 
本节从技术实现、经济成本与法律合规三个核心维度，对本项目设计方案的
可行性进行系统性论证。 
3.1 技术可行性 
本项目的技术可行性建立在大语言模型技术的成熟度与游戏引擎生态的完
备性之上。 
就大语言模型技术而言，以GPT-4、Claude等为代表的主流模型已在自然语
言理解、角色扮演及代码生成等任务上展现出稳定可靠的能力表现。在角色扮演
场景中，通过系统提示词与上下文管理策略，LLM能够维持一致的角色人格并生
成符合设定的对话内容，这一能力已在Character.AI等商业产品中得到验证。 
在代码生成层面，GitHub Copilot、Cursor等工具的广泛应用表明，LLM已
具备根据自然语言描述生成功能正确代码的能力。此外，主流服务商均已提供商
业化API接口，支持Function Calling等高级特性，为技术集成提供了便利条
件。在游戏引擎层面，Unity作为业界主流的跨平台开发引擎，具备成熟的工具
链与丰富的插件生态。 
针对动态Lua脚本执行需求，xLua等开源桥接库已在大量商业项目中得到验
证，支持C#对象向Lua环境的属性暴露，为LLM生成的卡牌脚本提供了可靠的
运行时支撑。就各功能模块而言，INFO与MEMORY模块属于常规的数据结构设计
问题，可通过LLM摘要能力结合轻量级向量数据库实现。AGENT模块的符号映射
机制本质上是字符串解析与函数路由，实现复杂度较低。 
PLANNER 模块可通过多轮对话与ReAct等提示词范式实现。自然语言驱动的
卡牌生成虽具创新性，但通过提供明确的模板结构与API文档，结合运行时校验
机制，能够保障功能稳定运行。 
3.2 经济可行性 
本项目的经济可行性涵盖开发工具成本、LLM服务支出及硬件资源需求三个
层面。在开发工具方面，Unity引擎提供免费的个人版授权，xLua等组件采用MIT
开源协议可免费使用，版本控制与编辑器等辅助工具同样提供免费版本。游戏美
术资源可通过Asset Store免费素材或开源资源库获取，有效控制制作成本。 
在LLM 服务成本方面，主流API采用按Token 计量的付费模式。以GPT-4o
为例，输入约$2.5/百万Token，输出约$10/百万Token。就典型使用场景估算，
单次对话交互消耗约1000-2000 Token，脚本生成约2000-4000 Token。 
开发阶段日均100次对话测试与20次脚本生成，月度成本约10-30美元，
处于可承受范围。此外，各服务商针对新用户提供免费额度，教育用户可申请研
究优惠，可进一步降低实际支出。在硬件资源方面，本项目采用云端API调用模

规渠道获取并依照许可协议使用。

益。

户数据收集范围与使用目的。玩家对话内容通过加密通道传输至LLM服务端，云
端处理符合服务商的隐私政策与数据处理协议。 
本地存储的用户数据采取适当加密措施保护。在内容安全方面，NPC对话系

执行，限制系统访问权限，防止恶意代码造成损害。整体设计符合国家内容管理
规定，确保生成内容不违反法律法规与社会公序良俗。
